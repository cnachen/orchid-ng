# Repository Guidelines

## Project Structure & Module Organization

This repository is a small Python package managed with `uv`. Application code lives under `src/orchid_ng/`; the current CLI entrypoint is [src/orchid_ng/cli/app.py](/Users/cnachen/Developer/orchid_ng/src/orchid_ng/cli/app.py). Package metadata and tool configuration live in [pyproject.toml](/Users/cnachen/Developer/orchid_ng/pyproject.toml). Design notes belong in [docs/](/Users/cnachen/Developer/orchid_ng/docs), and automated tests should be added under [tests/](/Users/cnachen/Developer/orchid_ng/tests) using the same module-oriented layout as `src/` when possible.

## Build, Test, and Development Commands

Use `uv` for local workflows:

- `uv sync --dev`: create/update the virtualenv with runtime and dev dependencies.
- `uv run orchid-ng`: run the installed CLI entrypoint.
- `uv run python -m orchid_ng`: run the package module directly.
- `uv run pytest`: run the test suite.
- `uv run ruff check .`: run lint checks.
- `uv run ruff format .`: apply the project formatter.
- `uv build`: build a distributable package from `pyproject.toml`.

## Coding Style & Naming Conventions

Target Python `>=3.12` and follow standard Python conventions: 4-space indentation, `snake_case` for modules/functions/variables, `PascalCase` for classes, and explicit imports from `orchid_ng`. Keep modules focused and prefer small service-style units over large mixed-responsibility files. Use Ruff for linting and formatting; run formatting before opening a PR.

## Testing Guidelines

`pytest` is the configured test framework. Add tests for every new behavior or bug fix, and keep them in files named `test_*.py`. Mirror the source layout where practical, for example `tests/cli/test_app.py` for CLI behavior. No coverage threshold is configured yet, so contributors should aim for meaningful path coverage on changed code.

## Commit & Pull Request Guidelines

Recent history follows short conventional prefixes such as `feat:`, `refactor:`, `chore:`, and `ckpg:`. Keep commit messages imperative and scoped to one change. PRs should include a brief summary, testing notes (`uv run pytest`, `uv run ruff check .`), and links to any related design note or issue. Include terminal output or screenshots only when the change affects CLI behavior or generated artifacts.

## Repository Hygiene

Do not commit virtualenv contents, caches, or generated bytecode such as `__pycache__/`. Keep `docs/` aligned with implementation changes when architecture or workflows shift.
